# 6일차 - Material

---
모델에 색을 입히기 위해서는 Fragment Shading, Texture Mapping 라는 절차가 더 필요하다.  
Fragment Shading은 픽셀마다 광량을 정해 3D모델을 렌더링 했을 떄 픽셀이 자연스러워 보이게 만드는 작업이다.  
Texture Mapping은 모델에 사진을 입혀 색깔을 칠하는 작업이다.   
우리는 지금까지 모델의 형태를 화면에 래스터라이징 하는데에 성공하였고,  
렌더러의 대략적인 기능을 모두 완성하기 위해서는 이 두 단계를 추가적으로 구현해야한다.  

프래그먼트와 텍스쳐를 구현하기 전에 생각해야 할 것이 있다. 색깔을 예쁘게 칠하기 위해서는 두 가지를 고민해야 한다.  
1. 대상의 물성
2. 사용할 텍스쳐  

우선, 물성에 대해 얘기하겠다. 물리 세계에는 여러 물질들이 있다. 표면이 매끈하면 빛을 정반사,  
반대로 거칠다면 빛을 난반사한다. 투명한 물질은 빛을 굴절시키기도 한다.  
이처럼 렌더링할 픽셀이 어떤 물성을 가지는지 안다면 더 예쁜 셰이딩 결과를 뽑아낼 수 있을 것이다.  

두 번쨰는 텍스쳐에 대한 이야기이다. 하나의 모델은 여러 개의 텍스쳐를 사용할 수 있다.  
예를 들어 연필을 든 학생을 예시로 들자면, 이 학생 모델은 연필, 옷, 피부 등의 텍스쳐를 하나의 이미지에 담는다면  
각 부위마다 서로 다른 물성을 하나의 텍스쳐 이미지로 제어해야하는 상황이 발생한다.

`.mtl`파일에는 위의 정보들이 기록되어 있다. 오늘은 이 정보들을 활용하기 위해 `.mtl` 파일의 정보들을 파싱하여  
렌더러에서 사용하 수 있도록 처리하는 Material 클래스를 만들어 보았다.

## Material.h

### struct material
우선 파싱된 정보를 보관할 구조체를 만들어야 한다.  

```markdown
newmtl BluePlastic
  Ka 0.1 0.1 0.1
  Kd 0.0 0.5 0.8
  Ks 0.5 0.5 0.5
  Ni 0.0 0.0 0.0
  Ke 0.0 1.0 0.0
  Ns 250.0
  d 1.0
  map_Kd textures/plastic.jpg

  illum 2

newmtl WoodBox
  Ka 1.0 1.0 1.0
  Kd 1.0 1.0 1.0
  Ks 0.0 0.0 0.0
  Ni 0.0 0.0 0.0
  Ke 0.0 1.0 0.0
  Ns 10.0
  d 1.0
  map_Kd textures/wood_diffuse.jpg

  illum 2
```
`.mtl`은 아래와 위와 같이 생겼다.  
`mtl`파일에는 각 태그에 따라 아래와 같은 값이 기록되어 있다.
- newmtl: 머터리얼의 이름
- Ka: Ambient Color
- Kd: Diffuse Color
- Ks: Specular Color
- Ni: Optical Density
- Ke: Emissive Color
- Ns: Specular Exponent
- d: Dissolved
- map_Kd: Path of Texture
- illum: 조명 모델  

뭐가 좀 많긴 한데 이런식이다. 

```c++
struct alignas(16) material {
    uint16_t materialId = 65535;
    uint16_t textureId = 65535;
    int16_t illumination = 0;
    uint16_t padding0 = 0;
    float dissolve = 1.f;
    float shininess = 0.f;

    Vec3 ambient = {0.2f, 0.2f, 0.2f};
    float opticalDensity = 1.f;

    Vec3 diffuse = {0.8f, 0.8f, 0.8f};
    float padding1 = 0;

    Vec3 specular = {0.f, 0.f, 0.f};
    float padding2 = 0;

    Vec3 emission = {0.f, 0.f, 0.f};
    float padding3 = 0;
};
```
위와 같이 구조체를 정의해 주었다. <[1일차 - Mesh](./1일차.md)>에서 나왔던 바와 같이 캐시 최적화를 위해 16바이트 간격으로 정렬해 주었다.  
이전에는 `alignas(16)`을 하면 알아서 16바이트 단위로 정렬해주고, 패딩을 자동으로 추가해준다고 설명하였는데  
이는 불완전한 설명이었다. 메모리 정렬을 통한 캐시 최적화를 위해 16바이트 단위의 정렬을 하는 이유는   
프로세서의 레지스터의 용량이 16바이트인 경우가 많기 때문이다.  
자주 같이 쓰이는 데이터를 16바이트 단위로 정렬해준다면 프로세서는 캐시에서 데이터를 다시 읽을 필요 없이 바로 다음 연산이 가능한 것이다.  
다만 주의해야 할 점은 하나의 데이터가 16바이트 단위 사이의 경계에 위치하지 않게 하는 것이다.  
4Byte의 데이터 하나가 캐시 상에서 1번째 단위와 2번째 나뉘어 있다면 프로세서는 이 데이터를 읽기 위해 확정적으로 2번의 캐시 읽기를 하게 된다.  
따라서 `padding` 변수를 추가하여 사용할 데이터들이 두개의 단위에 나뉘어 저장되는것을 막는 것이다.  
첨언하자면 이 구조체의 총 용량은 80Byte인데 가능하다면 용량을 64Byte로 맞춰준다면 메모리에서 캐시로 데이터를 읽어올 때  
캐시의 용량인 64Byte 안에 쏙 들어오니 더 좋다. 

고작 소프트웨어 렌더러 하나 만드는 주제에 쓸데없이 자잘한 최적화 때문에 서론이 길었다.  
구조체에 대한 설명을 하자면 위에서 설명한 `.mtl`파일 구조에 맞는 데이터를 담기 위한 변수들을 선언해 두었다.  
필자 보인도 그래픽스에 대한 조예가 깊지 않아 기본값 같은 경우는 AI의 도움을 받아 넣어주었다.  
여기서 머터리얼의 이름, 텍스쳐 파일의 경로를 `string`으로 적지 않고 ID로 만들어 정수값으로 선언한 이유는  
`stirng`타입은 크기가 가변적이어서 16바이트 단위 정렬이 불가능하기 때문이다.

### class Material

```c++
class Material {
public:
    bool loadMtl(const std::string &filename);
    [[nodiscard]] const material& get(const std::string &name) const {
        if (const auto it = mtls.find(name); it != mtls.end()) return it->second;
        static const material default_material;
        return default_material;
    }
private:
    std::map<std::string,material> mtls;
    std::vector<std::string> texturePaths;
};

```
머터리얼 정보를 저장하는 객체이다.  
`loadMtl()`은 `.mtl`파일의 경로를 입력받아 파일을 읽어 파싱한다.  
파싱한 머터리얼 데이터들은 맵 `mtls`에 저장된다.
`material`구조체는 텍스쳐 경로를 직접 저장하지 않고 정수 ID를 생성하여 저장한다.  
따라서 실제 테스쳐 경로를 저장해줄 `string` 배열이 필요하여 `texturePaths`를 만들어 주었다.  


## Material.cpp
구현부에는 `loadMtl()`의 로직만 분리해 주었다.  
```c++
bool Material::loadMtl(const string &filename) {
    unordered_map<string, uint16_t> pathMap;
    for (size_t i=0; i< texturePaths.size(); i++) {
        pathMap[texturePaths[i]] = static_cast<uint16_t>(i);
    }
    
    // 파싱에 필요한 스트림 변수들 정의
    
    while (getline(in, line)) {
        if (line.empty() || line[0] == '#') continue;

        istringstream iss(line.c_str());
        if (line.compare(0, 7, "newmtl ") == 0) {
            if (!first) {
                mtls[currName] = currMtl;
            }else first = false;
            currMtl = material();
            iss >> trash >> currName;
            currMtl.materialId = static_cast<uint16_t>(mtls.size());
            currMtl.textureId = numeric_limits<uint16_t>::max();
        }else if () {
        
            // 여러가지 속성들 처리
            
        }else if (line.compare(0, 7, "map_Kd") == 0) {
            string texturePath;
            iss >> trash >> texturePath;

            if (const auto it = pathMap.find(texturePath); it != pathMap.end()) {
                currMtl.textureId = it->second;
            }else {
                const auto id = static_cast<uint16_t>(texturePaths.size());
                texturePaths.push_back(texturePath);
                pathMap[texturePath] = id;
                currMtl.textureId = id;
            }
        }
    }
    if (currName.empty()) return false;
    mtls[currName] = currMtl;
    return true;
}
```
`Mesh.cpp`의 `loadObj()`와 비슷하게 늘 먹던 맛의 코드이다.  
파일을 한 줄씩 읽으면서 "newmtl" 발견되면 이전까지의 파싱 데이터를 `mtls`에 저장하고, 새로운 머터리얼을 정의하는 식이다.  
특이사항은 "map_Kd"태그를 파싱할 때 중복 검사 로직이 들어간다는 것이다.  
만약 서로 다른 머터리얼에서 같은 경로의 텍스쳐 파일을 사용한다면 `string` 데이터가 중복 저장되므로 메모리 낭비가 생긴다.  
(물론 디렉토리 아주 구석탱이에 텍스쳐 파일이 있는게 아니라면 괜찮을거다.)  
따라서 텍스쳐 파일 경로를 읽을 때 해시 테이블 `pathMap`을 통해 중복값이 있는지 비교하고 있다면 그 텍스쳐의 ID를 기록한다.  

## Mesh.h & Mesh.cpp

```c++
struct MeshGroup {
        uint32_t indexStart;
        uint32_t indexCount;
        uint16_t mtlId;
        std::string mtlName;
    };
```
`Mesh`에도 변경 사항이 있다. `MeshGroup`에 `mtlId`필드를 추가하여  
머터리얼의 ID를 정수값을고 가지고 있을 수 있게 만들었다.

```c++
void Mesh::linkMtl(const Material &mtls) {
    for (auto& group : subMeshes) {
        group.mtlId = mtls.get(group.mtlName).materialId;
    }
}
```
이어서 `Mesh`마다 `Material`을 연결해주는 함수를 추가하였다.
